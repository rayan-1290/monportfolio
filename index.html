Documentation des compétences — Projet Seyrin

C1

Recenser et identifier les ressources numériques

Réalisation : Recensement d'un parc informatique avec GLPI dans le cadre du Hackathon Seyrin.

Contexte

Dans le cadre d'un hackathon, nous avons mis en place un outil de gestion de parc informatique afin de centraliser les informations des équipements et des utilisateurs. L'objectif était d'avoir une vision claire du parc informatique grâce à GLPI.

Démarche suivie

  • Installation de GLPI sur un serveur Linux : installation des dépendances (Apache, PHP, MariaDB), démarrage des services et création de la base de données dédiée
  • Téléchargement et déploiement de GLPI dans /var/www/html, puis finalisation de l'installation via l'interface web
  • Installation de l'agent GLPI sur les postes pour remonter automatiquement les informations matérielles et logicielles vers le serveur
  • Configuration de l'agent : ajout de l'URL du serveur GLPI dans /etc/glpi-agent/agent.cfg, activation et démarrage du service
  • Création des comptes utilisateurs pour chaque membre de l'organisation (Amadou, Ézékiel, Johan, Kylian, Rayan, etc.)
  • Attribution des rôles à chaque compte (administrateur, technicien, utilisateur) pour gérer les droits d'accès et les responsabilités
  • Organisation du parc grâce aux remontées de l'agent : identification des machines (debian, fourmi-VMware, pc-de-rayan, pc-de-johan…), de leurs caractéristiques (processeur, OS, fabricant) et des logiciels installés (4.7K logiciels inventoriés)
# Installation des dépendances
$ sudo apt install apache2 php php-mysql php-curl php-gd php-xml mariadb-server
 
# Création de la base de données GLPI
$ sudo mysql -u root -p
CREATE DATABASE glpi;
CREATE USER 'glpi'@'localhost' IDENTIFIED BY 'motdepasse';
GRANT ALL PRIVILEGES ON glpi.* TO 'glpi'@'localhost';
 
# Installation et configuration de l'agent GLPI
$ sudo apt install glpi-agent
$ sudo nano /etc/glpi-agent/agent.cfg
server = http://IP_SERVEUR/glpi
$ sudo systemctl enable glpi-agent && sudo systemctl start glpi-agent

Compétences acquises

  • Installer et configurer GLPI ainsi que son agent sur plusieurs machines Linux
  • Recenser et identifier les ressources numériques (matérielles et logicielles) d'une organisation
  • Structurer un parc informatique en attribuant des rôles et en centralisant les informations
  • Exploiter les données d'inventaire remontées automatiquement par l'agent GLPI

C7

Collecte, suivi et orientation des demandes

Réalisation : Gestion des tickets d'incidents et des demandes d'assistance via GLPI pendant le Hackathon Seyrin.

Contexte

Dans le cadre d'un hackathon réalisé en groupe de 5 personnes à l'Efrei, notre équipe a été amenée à déployer et administrer une infrastructure informatique complète pour le site Seyrin. Nous avons utilisé GLPI pour gérer les tickets d'incidents et les demandes d'assistance émises par les membres de l'équipe tout au long du hackathon. Le tableau de bord GLPI était hébergé sur Azure (seyrin-glpi.francecentral.cloudapp.azure.com).

Cycle de vie d'un ticket

  • Nouveau — Le ticket est créé par un membre du groupe avec la description du problème
  • En cours (Attribué) — Le ticket est assigné au technicien compétent (Infra ou Support)
  • En cours (Planifié) — Le technicien analyse et travaille sur la résolution
  • Résolu — La solution est appliquée et documentée dans le ticket
  • Clôturé — Le demandeur confirme la résolution, le ticket est fermé

Exemples de tickets traités

  • Ticket #1 — Agent GLPI ne remonte pas l'inventaire : L'URL dans agent.cfg pointait en HTTP au lieu de HTTPS et le port 443 était fermé. Correction de l'URL, ouverture du port 443 via ufw, redémarrage du service. Inventaire remonté avec succès.
  • Ticket #2 — Accès refusé à l'interface GLPI : Un compte utilisateur avait été créé sans profil d'accès assigné. Attribution du profil « Technicien » depuis l'administration GLPI. Connexion vérifiée et fonctionnelle.
  • Ticket #3 — Machine absente de l'inventaire après redémarrage : Le service glpi-agent n'était pas configuré pour démarrer automatiquement au boot. Activation avec systemctl enable, vérification de la remontée d'inventaire.
# Correction URL agent GLPI
$ sudo nano /etc/glpi-agent/agent.cfg
server = https://IP_SERVEUR/glpi
 
# Ouverture du port 443
$ sudo ufw allow 443/tcp
$ sudo systemctl restart glpi-agent
 
# Activation démarrage automatique de l'agent
$ sudo systemctl enable glpi-agent

Compétences acquises

  • Maîtriser un logiciel de gestion de parc et d'incidents (GLPI)
  • Collecter, suivre et orienter des demandes via un système de ticketing
  • Diagnostiquer et résoudre des incidents de manière méthodique
  • Administrer les comptes utilisateurs avec des droits différenciés

C11

Référencement des services en ligne et mesure de leur visibilité

Réalisation : Mise en place de Google Analytics sur le site vitrine Seyrin Agency hébergé sur GitHub Pages.

Contexte

Dans le cadre du projet Hackathon, notre équipe Seyrin disposait d'un site vitrine hébergé sur GitHub Pages. L'objectif était de mettre en place un outil de mesure pour évaluer la visibilité du site en ligne et analyser le comportement des visiteurs grâce à Google Analytics.

Démarche suivie

  • Création d'un compte Google Analytics : connexion à analytics.google.com, création d'une propriété pour le site vitrine
  • Configuration du flux de données web avec l'URL du site GitHub Pages et récupération de l'identifiant de mesure (format G-XXXXXXXXXX)
  • Intégration du script de suivi Google Analytics (gtag.js) dans le <head> de chaque page du site vitrine
  • Vérification du bon fonctionnement : accès depuis plusieurs navigateurs et appareils, contrôle en temps réel dans Google Analytics, vérification de l'absence d'erreurs en console
<!-- Google tag (gtag.js) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXXXX');
</script>

Indicateurs analysés

  • Nombre de visiteurs sur une période donnée
  • Pages les plus consultées (Accueil, Équipe, Documentation)
  • Sources de trafic : accès direct, moteur de recherche, réseaux sociaux
  • Durée moyenne des sessions et taux de rebond (pourcentage de visiteurs quittant le site après une seule page)

Compétences acquises

  • Créer et configurer une propriété Google Analytics
  • Intégrer un script de suivi sur un site statique hébergé sur GitHub Pages
  • Analyser les indicateurs de fréquentation et de visibilité d'un site web
  • Prendre des décisions d'amélioration basées sur des données concrètes

C17

Déployer et rendre accessible un service informatique en ligne

Réalisation : Déploiement complet et sécurisé d'une infrastructure GLPI sur Microsoft Azure.

Contexte

Après une première installation locale de GLPI, il était nécessaire de le rendre accessible depuis l'extérieur afin que toute l'équipe puisse l'utiliser à distance. De plus, le fonctionnement initial en HTTP représentait un risque pour la confidentialité des données échangées. L'objectif était triple : acquérir et configurer un nom de domaine, déployer le serveur dans un environnement cloud fiable, et sécuriser les communications via HTTPS.

Architecture

  • Machine virtuelle Linux (Ubuntu Server) hébergée sur Microsoft Azure dans un groupe de ressources dédié
  • Serveur web Apache2 (ports 80 et 443)
  • Nom de domaine avec entrées DNS pointant vers l'IP publique Azure
  • Certificat SSL/TLS gratuit Let's Encrypt via Certbot

Démarche suivie

  • Étape 1 — Infrastructure cloud : Création d'un groupe de ressources Azure, déploiement d'une VM Ubuntu Server, configuration du NSG (ports 22, 80, 443), association d'une IP publique statique
  • Étape 2 — Serveur web : Connexion SSH à la VM, installation d'Apache et des modules PHP, transfert des fichiers GLPI via SCP vers /var/www/html/
  • Étape 3 — DNS : Enregistrement du nom de domaine, création des entrées DNS de type A (@ , www, glpi) pointant vers l'IP publique Azure
  • Étape 4 — HTTPS : Installation de Certbot, génération du certificat SSL Let's Encrypt, configuration du VirtualHost Apache avec les directives SSL, redirection automatique HTTP → HTTPS
# Ouverture des ports Azure
$ az vm open-port --port 80 --resource-group MonGroupe --name MaVM
$ az vm open-port --port 443 --resource-group MonGroupe --name MaVM
 
# Installation serveur web
$ sudo apt install -y apache2 php php-mysql php-curl php-gd php-xml php-mbstring
 
# Transfert des fichiers GLPI
$ scp -r ./glpi/ azureuser@IP_PUBLIQUE_VM:/var/www/html/
 
# Certificat SSL Let's Encrypt
$ sudo apt install -y certbot python3-certbot-apache
$ sudo certbot --apache -d monsite.fr -d glpi.monsite.fr

Tests et validation

  • Validation DNS avec nslookup et dig, vérification de la propagation mondiale via un outil en ligne
  • Test du certificat SSL dans le navigateur (cadenas valide, pas d'avertissement)
  • Test de renouvellement automatique : sudo certbot renew --dry-run
  • Accès depuis des postes externes (4G, Wi-Fi tiers) pour confirmer la disponibilité de GLPI

Compétences acquises

  • Déployer et configurer une infrastructure cloud sur Microsoft Azure (groupes de ressources, NSG, IP publiques)
  • Installer et configurer un serveur web Apache en environnement de production
  • Mettre en œuvre la gestion DNS et associer un nom de domaine à une infrastructure cloud
  • Sécuriser les communications via HTTPS avec Let's Encrypt et Certbot
